# 의사결정 기록 (Decision Log)

> ADR(Architecture Decision Record) 스타일. 코드만 봐선 알 수 없는 "왜 이렇게 했는가"를 기록.
> append-only. 결정을 번복할 경우 새 항목으로 추가하고 기존 항목에 "Superseded by D-XXX" 표시.

---

## D-001: Evolve 방식 (전면 재작성 아님)
- **일자**: 2026-05-17
- **결정**: 기존 코드 데이터 레이어(API 라우트, 데이터 처리 함수)는 보존. 시각 레이어(HTML, CSS, DOM 렌더링)만 재작성.
- **이유**: API 계약과 데이터 변환 로직이 이미 안정적이고, 향후 리포트도 동일 스키마를 따를 것임. 시각 레이어만 재작성하면 위험 감소 + 작업량 감소.
- **대안 검토**: 전면 재작성(scrap & rebuild) → 안정적인 부분까지 재작성하는 비용 과다, 리스크 증가
- **영향 범위**: 전체 Phase A~F. `server.mjs`의 데이터 헬퍼(`extractMetrics`, `formatChange`, `computeChange`, `fetchSupabaseReport`)와 API 라우트는 변경 안 함.

## D-002: 한국 금융 색상 관례 채택
- **일자**: 2026-05-17
- **결정**: 상승 = 빨강(`#d92d20`), 하락 = 파랑(`#1570ef`). 미국식(green up / red down)과 반대.
- **이유**: 한국 모든 금융 단말기·뉴스·증권사 리서치가 이 관례. 사용자(자금운용본부) 직관과 일치. 카카오톡 공유 시 받아보는 동료들도 동일 관례 기대.
- **대안 검토**: 미국식 색상 → 한국 사용자에게 혼동 유발
- **영향 범위**: CSS 토큰 `--up`, `--down`. JS의 색상 분기 로직. 헤더 핵심지표 칩, 데이터 표 행, 스파크라인 마지막 점 색.

## D-003: 스파크라인 외부 라이브러리 미사용
- **일자**: 2026-05-17
- **결정**: Chart.js / D3 / sparklines.js 등 미사용. 인라인 SVG `<polyline>` 자체 구현.
- **이유**: 36개 지표 × 7일 데이터로 단순한 추세선만 필요. 외부 의존성 추가 시 번들 크기·CSP·CDN 의존성 리스크. 인라인 SVG 50줄 이하로 충분.
- **대안 검토**: Chart.js (12KB) — 기능 과잉, sparklines.js — 유지보수 활발도 낮음
- **영향 범위**: Phase E. `server.mjs::buildReviewHtml` 끝부분 클라이언트 부트 스크립트.

## D-004: design.md 전면 교체
- **일자**: 2026-05-17
- **결정**: 기존 Notion 마케팅 사이트 디자인 시스템 문서 전체 삭제, 금융 대시보드용 새 문서로 교체.
- **이유**: Notion 마케팅 디자인은 1280px max-width + 80px 히어로 + 96px 섹션 간격 등 호흡 긴 마케팅 페이지용. 금융 데일리 리포트는 정보 밀도가 생명 — 정반대 철학. 일부 토큰 교체로는 부족하고 시스템 자체를 바꿔야 함.
- **대안 검토**: 토큰만 부분 교체 → CSS는 바뀌어도 디자인 문서가 잘못된 가이드를 주므로 다음 세션에서 다시 잘못된 방향으로 갈 가능성
- **영향 범위**: Phase A. `design.md` 전체.

## D-005: AI 질문창을 하단 고정 컴팩트 바로 이동
- **일자**: 2026-05-17
- **결정**: 현재 화면 중앙에 큰 영역으로 노출된 Q&A 섹션을 → 우하단 40px 접힘 컴팩트 바로 변경. 클릭 시 320px 패널로 펼침.
- **이유**: AI 답변은 보조 기능이지 주된 콘텐츠가 아님. 주된 콘텐츠인 시장 데이터 가시성을 침해하지 않아야 함. Bloomberg 터미널 같은 금융 도구가 표준 패턴.
- **대안 검토**: 사이드바 고정, 모달 — 화면 폭 차지 또는 흐름 끊김
- **영향 범위**: Phase D. 3개 페이지(admin / report / archive) 공통.

## D-006: 모든 Phase 한 번에 진행, PNG/PDF는 차기
- **일자**: 2026-05-17
- **결정**: Phase A~F를 한 번에 진행하는 큰 리디자인. PNG/PDF 내보내기(Phase G)는 1차 범위 제외, 차기 진행.
- **이유**: 사용자가 "큰 리디자인" 선택. 작은 단위로 끊으면 Notion 디자인 시스템이 남아서 일관성 깨짐. 다만 PNG/PDF는 모바일 반응형이 잘 되면 카카오톡 링크 공유로 대체 가능, 우선순위 낮음.
- **대안 검토**: 점진적 적용 → 중간 상태가 어색, 사용자 명시 반대
- **영향 범위**: 작업 순서. Phase G는 본 리디자인 안정화 후 별도 작업.

## D-007: 3열 카테고리 그리드 채택 (단일 테이블 X)
- **일자**: 2026-05-17
- **결정**: 8개 카테고리 36개 지표를 3열 그리드로 재배치. 열 1=국내(11), 열 2=해외 금리·주식(12), 열 3=외환·원자재·암호(12).
- **이유**: 사용자 요구 "한 페이지 안에 한눈에 보는 대시보드". 단일 컬럼 테이블은 위아래 스크롤 길어짐. 3열 균형 분배로 한 뷰포트 내에 거의 모든 지표 표시 가능.
- **대안 검토**: 단일 카테고리 테이블 (Plan agent 초기 추천) — 카테고리당 최대 7개 행이라 가로 폭 활용 안 됨; 2열 그리드 — 모바일 변환 시 어색
- **영향 범위**: Phase B. `buildReviewHtml` HTML 빌더 구조.

## D-008: 추적 인프라 4종 파일 신설 (메모리 의존 X)

- **결정**: 진행 상황을 메모리·세션 상태가 아닌 파일에 실시간 기록. `docs/STATUS.md` + `docs/REDESIGN_LOG.md` + `docs/DECISIONS.md` + `AGENTS.md` 현재 진행 섹션.

- **영향 범위**: 전체 작업 흐름. 매 작업 chunk마다 파일 갱신 의무.
